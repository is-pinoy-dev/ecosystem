This guide walks you through registering an is-pinoy.dev subdomain for a site hosted on GitHub Pages, then configuring GitHub Pages so it serves a valid HTTPS certificate for your subdomain.

## Prerequisites

- Your site is already deployed to GitHub Pages (e.g. `https://your-github-username.github.io`)
- You have chosen a subdomain (e.g. `yourname.is-pinoy.dev`)

---

<Steps>
  <Step>
    ## Find your CNAME value

    Your CNAME value is your GitHub Pages hostname:

    ```
    your-github-username.github.io.
    ```

    This is the same value for both user sites (`your-github-username.github.io`) and project sites (`your-github-username.github.io/my-project`) — the repository path is never part of the CNAME.

    <Callout type="warn">
      Keep the trailing dot (`.`) in your JSON file. This is standard DNS notation for a fully-qualified domain name. Replace `your-github-username` with your own GitHub username, lowercased.
    </Callout>

  </Step>
  <Step>
    ## Create your subdomain JSON file

    Create a file at `subdomains/yourname.json` in your fork of the repository, then follow the main [registration steps](/guides/workflow) to open a pull request.

    For reference, here is what your JSON should look like:

    ```json
    {
      "$schema": "https://raw.githubusercontent.com/is-pinoy-dev/ecosystem/main/packages/schemas/schema/v1/subdomain.schema.json",
      "subdomain": "yourname",
      "owner": {
        "github": "your-github-username"
      },
      "records": {
        "CNAME": {
          "value": "your-github-username.github.io."
        }
      }
    }
    ```

    Replace `yourname` and `your-github-username` with your actual information.

    <Callout type="warn">
      Do not set `"proxied": true` for GitHub Pages. The record must stay DNS-only so GitHub can verify your domain and issue the HTTPS certificate.
    </Callout>

  </Step>
  <Step>
    ## Add the custom domain in your Pages settings

    Do this after your PR is merged and the DNS record is live.

    1. Go to the repository that serves your Pages site (e.g. `your-github-username.github.io`)
    2. Click **Settings**, then **Pages** in the left sidebar
    3. Under **Custom domain**, enter your full subdomain:

    ```
    yourname.is-pinoy.dev
    ```

    4. Click **Save**

    GitHub will run a DNS check and then start provisioning a TLS certificate for your subdomain.

    <Callout type="warn">
      This step is required. Until you add the custom domain here, GitHub serves its default `*.github.io` certificate for your subdomain, and browsers will show an invalid certificate error when visiting `https://yourname.is-pinoy.dev`.
    </Callout>

  </Step>
  <Step>
    ## Wait for the certificate to be provisioned

    After saving the custom domain, the Pages settings page shows the certificate status (for example, "TLS certificate is being provisioned"). This usually completes within an hour, but can take up to 24 hours.

    If it seems stuck after 24 hours, remove the custom domain, save, then add it back to re-trigger provisioning.

  </Step>
  <Step>
    ## Enable Enforce HTTPS

    Once the certificate is issued, the **Enforce HTTPS** checkbox in **Settings** → **Pages** becomes clickable. It stays greyed out while the certificate is provisioning.

    Tick it. This makes GitHub redirect all `http://` visitors to `https://` automatically.

  </Step>
</Steps>

## Important notes

- **Order matters** — the DNS record from your merged PR must be live before GitHub can verify the custom domain and issue a certificate. If you added the custom domain before your PR was merged, remove it and add it again.
- **DNS-only is required** — do not enable `proxied: true` for GitHub Pages. GitHub needs to see the DNS record directly to verify the domain and provision HTTPS.
- **Invalid certificate errors** — these almost always mean the custom domain has not been added in your Pages settings yet, or the certificate is still provisioning.
- **CNAME file** — GitHub commits a file named `CNAME` containing your subdomain to the branch that serves your site. This is normal; do not delete it. If you deploy with GitHub Actions or a static site generator, make sure your build keeps a `CNAME` file in the published output.
- **One custom domain per repository** — the custom domain applies to the specific repository serving the site. Project sites need their own custom domain configured in their own repository's Pages settings.
- **DNS propagation** — DNS usually updates a few minutes after merge, but some resolvers can take up to 48 hours.

---

Once your site loads at `https://yourname.is-pinoy.dev` with a valid certificate, you're done. If you run into trouble, see GitHub's guide to [securing your GitHub Pages site with HTTPS](https://docs.github.com/en/pages/getting-started-with-github-pages/securing-your-github-pages-site-with-https).
